## 💡 How Can I Contribute?

### 🔍 Reporting Issues
- Check the [issue tracker](../../issues) to avoid duplicates.
- Use the appropriate template (bug report / feature request).
- Include: steps to reproduce, expected vs. actual behavior, environment details.

### ✨ Suggesting Features
- Open a discussion in [Discussions](../../discussions) first.
- Describe the problem, proposed solution, and potential impact.
- Be open to feedback and iteration.

### 🛠 Submitting Code
1. Fork the repository.
2. Create a feature branch: `git checkout -b feat/your-feature-name`.
3. Make your changes with clear, atomic commits.
4. Push to your fork and open a Pull Request.

### 📚 Improving Documentation
- Fix typos, clarify explanations, add examples.
- Documentation PRs are always welcome — no ticket required.

## 🧱 Coding Guidelines

### General
- Write clear, readable code — comments for "why", not "what".
- Keep functions small and focused.
- Prefer explicit over implicit.

### Style
- **Frontend**: ESLint + Prettier config (coming soon).
- **Backend**: Follow language-specific idioms (Python/Node.js — TBD).
- **Commits**: Use [Conventional Commits](https://www.conventionalcommits.org/):
  ```
  feat: add ZK-proof verification endpoint
  fix: handle expired credentials in cache
  docs: update API spec for /attest/habr
  ```

### Testing
- Add tests for new features / bug fixes.
- Ensure existing tests pass before submitting a PR.

## 🔄 Pull Request Process

1. **Pre-PR**: 
   - Ensure your branch is up-to-date with `main`.
   - Run tests and linting locally.

2. **PR Description**:
   - Link to related issue(s): `Closes #123`.
   - Describe what changed and why.
   - Include screenshots/UI changes if applicable.

3. **Review**:
   - At least one maintainer approval required.
   - Address feedback promptly; mark resolved threads.

4. **Merge**:
   - Squash & merge for feature branches.
   - Maintainers will handle deployment.
